What this report shows. How many virtual machines each storage configuration can host before its writes get too slow.

Each VM runs an identical workload — a steady stream of small random reads and writes, with each VM driving 500 IOPS. We add more VMs in steps; at each step we measure write latency across every VM and check the p99 write latency: the time taken by the slowest 1 % of writes (a number much more sensitive to real-world slowdowns than an average, because the slow tail is what users actually notice).

The capacity ceiling is the largest VM count where that p99 still stays under 5 ms. Push past it and some writes start taking many milliseconds longer than they should — which in production shows up as application slowdowns and timeouts.

3 configurations across 2 platforms: ROKS rep3-virt, ROKS rep2, vSAN raid5-ftt1.

All ramps tested at the same offered load (500 IOPS/VM) and the same SLA threshold (p99 < 5.0ms). Comparison is apples-to-apples.
How to read this report
  1. Comparability banner (above) — green when all ramps used the same offered load and SLA threshold (apples-to-apples); amber when any disagree (numbers may not be directly comparable).
  2. Capacity scorecard — one row per tier. Capacity is the last VM count that stayed under SLA. Breach is the first VM count that didn't. Resource ceiling = yes means the ramp stopped because the platform ran out of room to schedule more VMs, not because storage saturated.
  3. Ramp overlay chart — each pool has one color. Solid lines are aggregate IOPS (left axis); dashed lines are write-p99 latency (right axis). Dots are green when the step passed SLA, red when it failed. The horizontal red dashed line marks the SLA threshold (drawn only when all ramps share one); thin vertical dashed lines mark each pool's capacity ceiling.
  4. Per-ramp data — full step-by-step data for each ramp, collapsed by default. Click to expand.
Methodology

Workload

mixed-70-30-rated.fio — 70 % random read / 30 % random write, 4 KiB blocks, queue depth 32, one fio job per VM, rate-capped to the offered IOPS shown in the scorecard. Each VM runs against a 10 GiB test file on its own dedicated PVC/VMDK.

Ramp strategy

  • Phase 1 (doubling): 1 → 2 → 4 → 8 → ... VMs until write-p99 breaches the SLA, or the platform runs out of capacity to schedule more VMs.
  • Phase 2 (backfill): linear steps between the last passing and first failing VM counts to pin the capacity ceiling precisely.

Per-step methodology fixes

  • Prefill: every VM sequentially writes the full test file before measurement starts, allocating all RBD / VMDK / NFS-share objects up front. Without this, first-write allocation cost (4 MiB object allocation on thin RBD; block allocation on vSAN) gets counted as steady-state IO latency.
  • Wall-clock sync barrier: all VMs in a step wait on a shared epoch before launching their measurement window. Eliminates first-mover penalty where an early-starting VM measures alone for tens of seconds during other VMs' boot/prefill.

Both ROKS and vSAN use the same workload, ramp strategy, and methodology fixes. The only intentional difference is the storage backend under test.
